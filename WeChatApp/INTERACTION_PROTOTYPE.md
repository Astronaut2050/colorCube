# 彩块叠叠微信小程序 PRD

## 1. 产品概述

### 1.1 产品名称

彩块叠叠

### 1.2 产品定位

彩块叠叠是一款轻量休闲的方块下落与合并消除类小游戏。玩家通过左右滑动控制方块落点，通过下滑加速下落，使相同颜色或等级的方块合并，并通过整行消除获得分数。

### 1.3 目标平台

- 微信小程序
- H5 预览环境，仅用于界面和基础交互验证

### 1.4 目标用户

- 喜欢轻量休闲小游戏的用户
- 希望短时间内完成一局游戏的用户
- 对简单规则、低学习成本、即时反馈类玩法感兴趣的用户

## 2. 产品目标

1. 用户进入小程序后，可以快速理解玩法并开始游戏。
2. 支持选择不同棋盘列数：4列、6列、8列。
3. 不同列数下，游戏区域宽高保持一致，行数保持一致，方块宽度随列数变化。
4. 支持基础触摸操作：左滑、右滑、下滑。
5. 支持暂停、继续、重新开始、退出游戏。
6. 支持无限模式和快速模式。
7. 微信前后台切换时，游戏状态稳定，不出现计时跳变、重复计时或重复下落。

## 3. 用户流程

```text
启动页
  -> 模式设置页
    -> 游戏中
      -> 暂停
        -> 继续游戏
        -> 重新开始
        -> 退出游戏
      -> 游戏结束
        -> 再来一局
        -> 退出游戏
      -> 挑战成功
        -> 再来一局
        -> 退出游戏
  -> 游戏指引
```

## 6. 游戏中页面

### 6.1 展示内容

- 当前得分
- 当前状态
- 快速模式计时器
- 指引按钮
- 暂停按钮
- 游戏棋盘
- 当前下落方块
- 下一方块预览

### 6.2 操作方式

| 用户操作 | 游戏行为 |
| --- | --- |
| 左滑 | 当前方块向左移动一列 |
| 右滑 | 当前方块向右移动一列 |
| 下滑 | 当前方块加速下落 |
| 点击暂停 | 暂停游戏 |
| 点击指引 | 暂停当前游戏并打开指引 |

### 6.3 触摸判定规则

```text
minimumDistance = 24px

if abs(deltaX) > abs(deltaY) and abs(deltaX) > minimumDistance:
  deltaX < 0 -> 左移
  deltaX > 0 -> 右移

if deltaY > minimumDistance and abs(deltaY) >= abs(deltaX):
  触发软降落
```

### 6.4 棋盘规格

游戏可操作区域宽高固定，行数固定，列数随用户设置变化。

```text
boardWidth = 可用游戏区域宽度
boardHeight = 可用游戏区域高度
rows = 固定可操作行数
cols = 用户选择列数

gapX = clamp(3, boardWidth * 0.015, 6)
gapY = clamp(3, boardHeight * 0.01, 6)

cellWidth = floor((boardWidth - gapX * (cols + 1)) / cols)
cellHeight = floor((boardHeight - gapY * (rows + 1)) / rows)
```

### 6.5 列数显示规则

| 列数 | 游戏区域 | 方块宽度 | 方块高度 | 行数 |
| --- | --- | --- | --- | --- |
| 4列 | 全宽显示 | 最大 | 按固定行数计算 | 不变 |
| 6列 | 全宽显示 | 中等 | 按固定行数计算 | 不变 |
| 8列 | 全宽显示 | 最小 | 按固定行数计算 | 不变 |

说明：

- 方块可以不是正方形。
- 横向移动按 `cellWidth` 计算。
- 纵向下落按 `cellHeight` 计算。
- 触摸命中区、动画距离、当前方块、已落方块必须使用同一套尺寸计算。
- 不允许因为列数变化导致棋盘宽度缩小或超出游戏区域。

## 7. 游戏规则

### 7.1 方块生成

- 每次生成一个新方块。
- 新方块从棋盘顶部生成。
- 新方块初始等级从低等级中随机产生。
- 下一方块需要在棋盘顶部以预览条或预览块形式展示。

### 7.2 自动下落

- 游戏开始后，当前方块按照所选速度自动下落。
- 速度等级越高，下落间隔越短。
- 暂停、游戏结束、挑战成功时，下落停止。

### 7.3 软降落

- 用户下滑时，当前方块进入短时间加速下落状态。
- 软降落结束后恢复正常速度。
- 暂停或游戏结束时必须清除软降落状态。

### 7.4 落地结算

当前方块满足以下任一条件时进入结算：

- 到达棋盘底部。
- 下方已有方块阻挡。

### 7.5 合并规则

- 当前方块落地后，如果正下方方块等级相同，则发生合并。
- 合并后，下方方块等级提升一级。
- 合并后增加分数。
- 最高等级方块是否继续合并，需要在实现阶段明确；默认可继续计分，但等级不再提升。

### 7.6 下落补位

- 方块合并或消除后，上方悬空方块继续向下补位。
- 补位过程中如果再次满足合并条件，则继续结算。
- 结算需要设置保护上限，避免异常死循环。

### 7.7 整行消除

- 当某一整行被填满，且该行所有方块等级相同时，触发整行消除。
- 消除后，上方方块整体下落补位。
- 消除后增加分数。

### 7.8 失败规则

- 新方块生成位置已被占用时，游戏结束。
- 游戏结束后停止游戏循环、停止计时、停止软降落。

### 7.9 快速模式胜利规则

- 快速模式下，当前得分达到或超过目标分数时，立即判定挑战成功。
- 若同一帧同时满足挑战成功和游戏结束，优先判定挑战成功。

## 8. 计分规则

### 8.1 合并计分

不同等级方块合并时，按等级增加分数。

建议分值：

| 等级 | 分值 |
| --- | --- |
| 1级 | 1分 |
| 2级 | 2分 |
| 3级 | 3分 |
| 4级 | 4分 |
| 5级 | 5分 |

### 8.2 整行消除计分

整行消除得分建议：

```text
lineScore = 当前列数 * 当前行等级分值
```

## 13. 微信生命周期

| 微信事件 | 当前游戏状态 | 预期行为 |
| --- | --- | --- |
| onHide | 游戏中 | 自动暂停游戏 |
| onShow | 自动暂停 | 自动恢复游戏 |
| onHide | 手动暂停 | 保持暂停 |
| onShow | 手动暂停 | 保持暂停 |
| onShow | 游戏结束 | 保持游戏结束 |
| onShow | 挑战成功 | 保持挑战成功 |

说明：

- 自动暂停和手动暂停需要区分。
- 从后台恢复时，需要重置时间基准，避免方块瞬间下落多格。
- 前后台切换不能创建重复计时器或重复动画循环。

## 14. 视觉与布局要求

### 14.1 整体风格

- 休闲、轻量、柔和。
- 以米色背景、深色棋盘、绿色主按钮为主要视觉风格。
- 页面优先服务游戏操作，不做营销式首页。

### 14.2 顶部栏

- 显示得分、状态、指引按钮。
- 游戏中显示暂停按钮。
- 长分数不得挤压按钮。
- 小屏设备上不得出现文字重叠。

### 14.3 游戏区域

- 游戏区域在不同列数下保持同宽同高。
- 棋盘不得横向溢出。
- 棋盘不得被底部安全区遮挡。
- 高 DPR 设备下 Canvas 必须清晰。

### 14.4 弹层

- 暂停、游戏结束、挑战成功均使用居中弹层。
- 弹层按钮宽度一致。
- 弹层不得遮挡顶部状态栏的关键反馈。

## 15. 技术实现要求

### 15.1 核心逻辑

游戏核心需要与平台解耦，不能直接依赖 DOM、Canvas 或微信 API。

建议核心接口：

```text
game.start(options)
game.pause()
game.resume()
game.moveLeft()
game.moveRight()
game.softDropStart()
game.softDropStop()
game.tick(timestamp)
game.getSnapshot()
```

### 15.2 渲染层

渲染层只消费游戏快照，不直接修改游戏状态。

建议渲染接口：

```text
renderer.resize(width, height, pixelRatio)
renderer.draw(snapshot, now)
renderer.destroy()
```

### 15.3 微信适配层

微信适配层负责：

- 页面生命周期转换为游戏命令。
- 触摸事件转换为游戏命令。
- Canvas 初始化。
- 安全区信息读取。
- 本地存储读取与写入。

1. 用户可以从启动页进入设置页。

11. Canvas 在高 DPR 设备上显示清晰。
